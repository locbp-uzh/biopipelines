## Machine Configuration and Installation

BioPipelines requires a Linux environment with conda or mamba, and optionally a SLURM-managed HPC cluster with GPU nodes. To install, clone the repository, create the base conda environment from the provided `Environments/biopipelines.yaml` file, and install the package in editable mode with `pip install -e .`. For Jupyter notebook usage, an IPython kernel can be registered. Each computational tool (e.g., RFdiffusion, AlphaFold, ProteinMPNN) must be installed separately into its own conda environment following the respective tool's upstream instructions; the environment names are then registered in `config.yaml`.

The `config.yaml` file (a minimal template is provided in `config_SI.yaml`) must be adapted to the target machine. It defines folder paths for tool repositories, shared storage, and cache directories, as well as the conda environment name for each tool and cluster-specific settings such as the environment manager (`mamba` or `conda`), container executor, and SLURM module loads. BioPipelines also supports Google Colab out of the box, where it auto-detects the environment, uses `pip` instead of conda, and runs tools on-the-fly without SLURM.

## Extending BioPipelines with New Tools

New tools are added by creating a Python module in `biopipelines/` that subclasses `BaseConfig`. Each tool must implement four methods: `validate_params()` for input validation, `configure_inputs()` to set up folder paths, `generate_script()` to produce the bash script that will run on the cluster, and `get_output_files()` to predict output file paths and metadata. Tools never execute computations directly — they generate bash scripts at configuration time, and these scripts are later executed on SLURM nodes. If a tool requires complex data processing, a companion runtime script (`HelpScripts/pipe_*.py`) is created, which is invoked by the generated bash script and can use the `biopipelines_io` module to load DataStreams and iterate over input structures.

Tools communicate through a standardized output system: each tool declares its outputs as `DataStream` objects (for files like PDB structures or sequences) and `TableInfo` objects (for CSV metadata). Downstream tools consume these outputs by accepting `DataStream` or `StandardizedOutput` parameters, enabling modular chaining without tools needing to know about each other. Multi-input combinatorics (cartesian products, bundling) and ID provenance tracking are handled automatically by the framework's `combinatorics.py` module, so tool authors only need to declare their input axes.
